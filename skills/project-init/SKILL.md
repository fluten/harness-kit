# project-init — 项目管理体系初始化 Skill

基于已有的 SPEC.md，生成项目管理体系的 .md 文件。不是每个项目都需要全部 9 个文件——按项目复杂度分层生成，避免管理开销超过项目本身。

---

## 触发条件

用户希望初始化项目管理文件，且项目中已有 SPEC.md。

## 前置检查

1. 确认 SPEC.md 存在且内容完整
2. 如果 SPEC.md 不存在，提示用户先生成 SPEC（可引导使用 spec-gen Skill）
3. 从 SPEC.md 中提取以下信息供后续文件使用：
   - 项目名、一句话描述
   - 技术栈
   - 功能模块列表
   - 数据需求（判断是否需要 SCHEMA.md）
   - 产品形态（判断是否需要 DESIGN.md）
   - 模块数量（判断是否需要 ARCHITECTURE.md）

---

## 三层生成策略

### 第一层：必生成（每个项目都要）

| 文件 | 说明 |
|------|------|
| CLAUDE.md | 项目管理体系入口，Claude Code 启动时自动读取 |
| TODO.md | 当前任务清单 |

### 第二层：初始化时按需生成（读 SPEC 自动判断）

| 文件 | 生成条件 | 不满足时 |
|------|----------|----------|
| ARCHITECTURE.md | SPEC 中功能模块 ≥ 3 个 | 不生成，模块关系在 CLAUDE.md 中简要描述 |
| SCHEMA.md | SPEC 中有数据库或复杂数据模型 | 不生成，数据结构在 SPEC.md 数据需求区描述 |
| DESIGN.md | SPEC 中产品形态有用户界面 | 不生成 |

条件不确定时，询问用户一次确认。

### 第三层：开发过程中按需生成

以下文件不在初始化时创建。CLAUDE.md 中预留这些文件的模板规则，当 Claude Code 在开发中遇到触发条件时，按模板自动创建并通知用户。

| 文件 | 触发条件 |
|------|----------|
| ISSUES.md | 开发中发现第一个 bug、技术问题或待决策事项时 |
| DECISIONS.md | 开发中遇到第一个需要做技术选型或架构决策的时刻时 |
| PROGRESS.md | 第一次跨 session 续接（新 session 开始时发现需要回顾上次进度）时 |

---

## 通用规则（适用于所有文件）

### 标注规则
每个文件必须包含：
1. **文件开头**：区块权限总表（HTML 注释内的 Markdown 表格），声明每个区块是锁定区还是生成区
2. **每个区块**：用 `<!-- 🔒 START 锁定区：xxx -->` / `<!-- 🔒 END 锁定区：xxx -->` 或 `<!-- 🔓 START 生成区：xxx -->` / `<!-- 🔓 END 生成区：xxx -->` 包裹
3. 决策文件在生成区标记中追加 `（⚠️ 决策文件：修改需用户确认）`

### LLM 可读性规则
1. 不使用任何 ASCII 图（架构图、ER 图、流程图、目录树等）
2. 所有关系、结构、流程用结构化文本描述（关系列表、扁平路径、箭头文本）
3. 不写模糊表述，必须具体

### 文件权限分类
- **过程文件**（TODO.md、ISSUES.md、PROGRESS.md）：AI 可自主维护
- **决策文件**（SPEC.md、ARCHITECTURE.md、DECISIONS.md、SCHEMA.md、DESIGN.md）：修改前必须获得用户确认

---

## 各文件模板规范

### CLAUDE.md（必生成）

项目管理体系的唯一入口，Claude Code 启动时自动读取。

区块结构：
1. 🔓 项目信息 — 从 SPEC 提取：项目名、描述、技术栈、仓库地址
2. 🔒 文件体系 — 锁定内容：文件表格（文件、何时读取、何时修改、权限）。已存在的文件正常列出；第三层文件标注"文件存在时才读取，触发条件见下方"
3. 🔒 自动化行为 — 锁定内容：
   - PROGRESS 更新规则（PROGRESS.md 存在时执行）
   - ISSUES 联动规则（ISSUES.md 存在时执行）
   - ISSUES 编号规则：类型-模块-序号，六种类型（BUG/DEBT/QUESTION/LIMIT/PERF/FEAT）
   - SPEC 拆分提醒：超过 300 行时提醒
   - 跨模块检查
   - DECISIONS 记录规则（DECISIONS.md 存在时执行）
4. 🔒 第三层文件自动创建规则 — 锁定内容：
   - 当发现第一个 bug/问题/待决策事项时 → 按模板创建 ISSUES.md 并通知用户
   - 当遇到第一个技术选型决策时 → 按模板创建 DECISIONS.md 并通知用户
   - 当新 session 开始且需要回顾上次进度时 → 按模板创建 PROGRESS.md 并通知用户
   - 每个文件的完整模板规则写在此区块内，确保 Claude Code 创建时有据可依
5. 🔓 代码规范 — 从 SPEC 技术栈推导：语言规范、命名规范、commit 格式、模块缩写约定表
6. 🔓 当前 Session 上下文 — 初始值：上次做到/当前目标/注意事项

### TODO.md（必生成）

区块结构：
1. 🔓 当前任务 — 从 SPEC 的 MVP 范围推导前 1-3 项
2. 🔓 待办队列 — 按高/普通/低优先级分组，从 SPEC 功能模块拆解
3. 🔓 已完成 — 初始为空
4. 🔒 规则 — 锁定内容：当前任务限 1-3 项、完成时同步 PROGRESS 和 ISSUES（如文件存在）、引用 issue 格式、已完成区按时间倒序

### ARCHITECTURE.md（初始化按需：模块 ≥ 3）

区块结构（全文为生成区，⚠️ 决策文件）：
1. 系统概述 — 一段话描述整体架构
2. 系统架构关系 — 结构化文本：`ComponentA --协议--> ComponentB`，明确禁止和允许的访问路径
3. 模块划分 — 每个模块：职责、对外接口、依赖、被依赖、关键文件路径
4. 模块依赖关系 — 文本描述依赖方向，标注是否有循环依赖
5. 数据流 — 按核心流程描述数据经过的模块链路
6. 目录结构 — 扁平路径列表（每行一个路径 + 说明）
7. 技术选型摘要 — 表格引用 DECISIONS.md 编号（如 DECISIONS.md 存在）
8. 模块间约定 — 通信方式、错误处理、命名约定
9. 扩展规划

### SCHEMA.md（初始化按需：有数据库）

区块结构（全文为生成区，⚠️ 决策文件）：
1. 数据概览
2. 表关系 — 结构化文本：`tableA 1:N tableB（通过 tableB.field_name）`
3. 表结构 — 每张表一个四列表格（字段、类型、约束、说明）
4. 索引
5. 枚举/常量
6. 数据约定 — 主键策略、时间字段、软删除、字符编码
7. 迁移记录
8. 🔒 规则

### DESIGN.md（初始化按需：有界面）

区块结构：
- Part 1 视觉规范（🔓 生成区，⚠️ 决策文件）：设计风格、色彩体系（主色/中性色/语义色）、字体、间距系统、圆角、组件规范
- Part 2 交互设计（🔓 生成区，⚠️ 决策文件）：页面结构（文本描述布局）、页面流（文本描述跳转路径）、状态与反馈（加载态/空状态/错误处理）、动效规范、响应式
- 🔒 规则

### ISSUES.md（开发中按需：遇到问题时）

区块结构：
1. 🔓 统计概览 — 四态数量表
2. 🔓 🔴待处理 / 🟡处理中 / 🟢已修复 / ✅已验证 — 四个状态分区
3. 🔓 历史归档 — 已验证超 10 条后移入
4. 🔒 规则 — 锁定内容：编号格式（类型-模块-序号）、六种类型（BUG/DEBT/QUESTION/LIMIT/PERF/FEAT）、四态流转（🔴→🟡→🟢→✅）、不同类型必填字段（BUG 要复现步骤、PERF 要指标、QUESTION 要方案列表、FEAT 要用户故事）、归档机制

### DECISIONS.md（开发中按需：遇到选型时）

区块结构：
1. 🔓 全局决策 + 各模块决策 — 每个区块下分 ✅已确定 / 💬待讨论 / 🗑️已废弃
2. 🔒 规则 — 锁定内容：ADR 编号格式（ADR-模块-序号）、三态分区、废弃留痕机制、不可逆程度必填（🟢易调整/🟡有成本/🔴不可逆）、禁止推翻🔴不可逆决策

### PROGRESS.md（开发中按需：跨 session 续接时）

区块结构：
1. 🔓 项目状态 — 🟡规划中 / 🔵开发中 / 🟢已上线 / 🔴已暂停
2. 🔓 最近完成 — 每完成一个 TODO 项追加
3. 🔓 当前阻塞 — 从 ISSUES 中提取阻塞项（如 ISSUES.md 存在）
4. 🔓 里程碑 — 从 SPEC 的 MVP 范围推导
5. 🔓 Session 日志 — 每次 session 结束写入
6. 🔒 规则 — 锁定内容：最近完成追加规则、超 20 条归档到 Session 日志、阻塞更新、里程碑状态、Session 日志三要素（完成了什么/遇到什么问题/下次从哪继续）

## 执行流程

1. 读取 SPEC.md，提取项目信息、模块列表、技术栈、数据需求、产品形态
2. 判断第二层文件的生成条件：
   - 模块 ≥ 3？→ 生成 ARCHITECTURE.md
   - 有数据库？→ 生成 SCHEMA.md
   - 有界面？→ 生成 DESIGN.md
   - 条件不确定时询问用户
3. 按以下顺序生成文件：
   - CLAUDE.md（含第三层文件的创建规则和模板）
   - TODO.md
   - ARCHITECTURE.md（如需要）
   - SCHEMA.md（如需要）
   - DESIGN.md（如需要）
4. 做一次交叉校验：
   - CLAUDE.md 的模块缩写表和 SPEC 的模块列表一致
   - ARCHITECTURE.md 的模块依赖和 SPEC 的模块描述一致（如生成了 ARCHITECTURE.md）
   - TODO.md 的任务覆盖了 SPEC 的 MVP 范围
5. 将所有文件放在项目根目录，创建 docs/spec/ 空目录
6. 告知用户：ISSUES.md、DECISIONS.md、PROGRESS.md 将在开发过程中按需自动创建
