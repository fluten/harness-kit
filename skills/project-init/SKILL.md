# project-init — 项目管理体系初始化 Skill

基于已有的 SPEC.md，生成项目管理体系的 .md 文件。不是每个项目都需要全部 9 个文件——按项目复杂度分层生成，避免管理开销超过项目本身。

---

## 触发条件

用户希望初始化项目管理文件，且项目中已有 SPEC.md。

## 前置检查

1. 确认 SPEC.md 存在且内容完整
2. 如果 SPEC.md 不存在，提示用户先生成 SPEC（可引导使用 spec-gen Skill）
3. 从 SPEC.md 中提取以下信息供后续文件使用：
   - 项目名、一句话描述
   - 技术栈
   - 功能模块列表
   - 数据需求（判断是否需要 SCHEMA.md）
   - 产品形态（判断是否需要 DESIGN.md）
   - 模块数量（判断是否需要 ARCHITECTURE.md）

---

## 三层生成策略

### 第一层：必生成（每个项目都要）

| 文件 | 说明 |
|------|------|
| CLAUDE.md | 项目管理体系入口，Claude Code 启动时自动读取 |
| TODO.md | 当前任务清单 |

### 第二层：初始化时按需生成（读 SPEC 自动判断）

| 文件 | 生成条件 | 不满足时 |
|------|----------|----------|
| ARCHITECTURE.md | SPEC 中功能模块 ≥ 3 个 | 不生成，模块关系在 CLAUDE.md 中简要描述 |
| SCHEMA.md | SPEC 中有数据库或复杂数据模型 | 不生成，数据结构在 SPEC.md 数据需求区描述 |
| DESIGN.md | SPEC 中产品形态有用户界面 | 不生成 |

条件不确定时，询问用户一次确认。

### 第三层：开发过程中按需生成

以下文件不在初始化时创建。CLAUDE.md 中预留这些文件的模板规则，当 Claude Code 在开发中遇到触发条件时，按模板自动创建并通知用户。

| 文件 | 触发条件 |
|------|----------|
| ISSUES.md | 开发中发现第一个 bug、技术问题或待决策事项时 |
| DECISIONS.md | 开发中遇到第一个需要做技术选型或架构决策的时刻时 |
| PROGRESS.md | 第一次跨 session 续接（新 session 开始时发现需要回顾上次进度）时 |

## 通用规则（适用于所有文件）

### 标注规则
每个文件必须包含：
1. **文件开头**：区块权限总表（HTML 注释内的 Markdown 表格），声明每个区块是锁定区还是生成区
2. **每个区块**：用 `<!-- 🔒 START 锁定区：xxx -->` / `<!-- 🔒 END 锁定区：xxx -->` 或 `<!-- 🔓 START 生成区：xxx -->` / `<!-- 🔓 END 生成区：xxx -->` 包裹
3. 决策文件在生成区标记中追加 `（⚠️ 决策文件：修改需用户确认）`

### LLM 可读性规则
1. 不使用任何 ASCII 图（架构图、ER 图、流程图、目录树等）
2. 所有关系、结构、流程用结构化文本描述（关系列表、扁平路径、箭头文本）
3. 不写模糊表述，必须具体

### 文件权限分类
- **过程文件**（TODO.md、ISSUES.md、PROGRESS.md）：AI 可自主维护
- **决策文件**（SPEC.md、ARCHITECTURE.md、DECISIONS.md、SCHEMA.md、DESIGN.md）：修改前必须获得用户确认

### 📊 快速状态 头块规范（适用于 TODO / ISSUES / PROGRESS / DECISIONS）

四个状态可变文件的顶部必须含一个**机械可重建的头块**，让 LLM 在读 body 之前就能判断"要不要读"。

**位置**：紧跟在区块权限声明之下，所有 body 区块之前
**性质**：🔓 生成区（AI 可写），但内容必须由 body 机械派生（不允许主观判断）
**形式**：bullet KV 列表，值用反引号包裹（LLM 易读）
**重建机制**：每个文件的 🔒 锁定区底部必须包含"重建算法"——一段确定性伪代码，告诉 AI "怎么从 body 数出每个字段"。AI 修改 body 后必须按算法**整体重写**头块（禁止增量修改）。

**四个文件的头块定位**：

| 文件 | 头块名 | 类型 | 字段 |
|---|---|---|---|
| TODO.md | 📊 快速状态 | triage | 进行中数 / 待办分布 / 下一步 / 更新 |
| ISSUES.md | 📊 快速状态 | triage | 活跃模块 / 起读位置 / 状态计数 / 阻塞 MVP / 更新 |
| PROGRESS.md | 📊 快速状态 | triage | 阶段 / 当前里程碑 / 阻塞计数 / 下次从 / 更新 |
| DECISIONS.md | 🔴 不可逆决策清单 | 禁区提醒 | 不可逆决策列表 / 数量 / 更新 |

**字段设计原则**（LLM 友好性）：
1. **纯机械派生**：每个字段都必须能从 body 用 count / scan / sort / extract 得到，禁止"重要性""健康度"这类主观判断
2. **bullet + 加粗键 + 反引号值**：`- **key**: \`value\`` 是 LLM 最容易解析的 KV 模式
3. **固定字段顺序**：跨项目永远保持同一顺序，让 LLM 学会模式
4. **绝对日期**：`YYYY-MM-DD` 而非"昨天/最近"
5. **整块 ≤ 10 行**：不抢 body 的首屏空间
6. **明确的 START/END 注释标记**：`<!-- 🔓 START 生成区：📊 快速状态 -->` 让 AI 知道精确边界，方便整体重写
7. **顶部带 🤖 注释**说明"AI 自动维护，必须按底部算法重建"

**重建算法的写法**：
- 写在每个文件的 🔒 锁定区（即"规则"区块）的最后一节
- 标题为 `### 📊 快速状态 重建算法`（DECISIONS 用 `### 🔴 不可逆决策清单 重建算法`）
- 结构：**输入 → 输出 → 步骤 → 输出模板**
- 步骤用编号列表，每步是 `count / scan / extract / filter` 的纯机械操作
- 输出模板用 4-backtick 代码块包裹，用 `{var}` 占位符标记要替换的值

**为什么不放在 SKILL.md 里**（option A）而是塞在每个模板里（option B）：
- skill 是一次性生成器，生成完就退场，不会被 AI 在日常开发中持续读取
- 算法必须刻在 AI 长期面对的文件里（即生成出来的 TODO.md 等），才能被持续执行
- 模板里的 token 成本（30-50 token/文件）远小于"AI 不知道有这条规则"的代价
- locality：算法和它要重写的目标在同一个文件里，AI 不需要跨文件查规则

具体的算法内容见各模板（`templates/*.example` 的 🔒 规则区块），生成时按模板一字不差地复制到目标项目即可。

## 各文件模板规范

### CLAUDE.md（必生成）

项目管理体系的唯一入口，Claude Code 启动时自动读取。

区块结构：
1. 🔓 项目信息 — 从 SPEC 提取：项目名、描述、技术栈、仓库地址
2. 🔒 文件体系 — 锁定内容：文件表格（文件、何时读取、何时修改、权限）。已存在的文件正常列出；第三层文件标注"文件存在时才读取，触发条件见下方"
3. 🔒 自动化行为 — 锁定内容：
   - PROGRESS 更新规则（PROGRESS.md 存在时执行）
   - ISSUES 联动规则（ISSUES.md 存在时执行）
   - ISSUES 编号规则：类型-模块-序号，六种类型（BUG/DEBT/QUESTION/LIMIT/PERF/FEAT）
   - SPEC 拆分提醒：超过 300 行时提醒
   - 跨模块检查
   - DECISIONS 记录规则（DECISIONS.md 存在时执行）
4. 🔒 第三层文件自动创建规则 — 锁定内容：
   - 当发现第一个 bug/问题/待决策事项时 → 按模板创建 ISSUES.md 并通知用户
   - 当遇到第一个技术选型决策时 → 按模板创建 DECISIONS.md 并通知用户
   - 当新 session 开始且需要回顾上次进度时 → 按模板创建 PROGRESS.md 并通知用户
   - 每个文件的完整模板规则写在此区块内，确保 Claude Code 创建时有据可依
5. 🔓 代码规范 — 从 SPEC 技术栈推导：语言规范、命名规范、commit 格式、模块缩写约定表
6. 🔓 当前 Session 上下文 — 初始值：上次做到/当前目标/注意事项

### TODO.md（必生成）

区块结构：
1. 🔓 **📊 快速状态** — 头块（详见上方「📊 快速状态 头块规范」），字段：进行中 / 待办（高/普通/低）/ 下一步 / 更新
2. 🔓 当前任务 — 从 SPEC 的 MVP 范围推导前 1-3 项
3. 🔓 待办队列 — 按高/普通/低优先级分组，从 SPEC 功能模块拆解
4. 🔓 已完成 — 初始为空
5. 🔒 规则 — 锁定内容：当前任务限 1-3 项、完成时同步 PROGRESS 和 ISSUES（如文件存在）、引用 issue 格式、已完成区按时间倒序、**📊 快速状态 重建算法**

### ARCHITECTURE.md（初始化按需：模块 ≥ 3）

区块结构（全文为生成区，⚠️ 决策文件）：
1. 系统概述 — 一段话描述整体架构
2. 系统架构关系 — 结构化文本：`ComponentA --协议--> ComponentB`，明确禁止和允许的访问路径
3. 模块划分 — 每个模块：职责、对外接口、依赖、被依赖、关键文件路径
4. 模块依赖关系 — 文本描述依赖方向，标注是否有循环依赖
5. 数据流 — 按核心流程描述数据经过的模块链路
6. 目录结构 — 扁平路径列表（每行一个路径 + 说明）
7. 技术选型摘要 — 表格引用 DECISIONS.md 编号（如 DECISIONS.md 存在）
8. 模块间约定 — 通信方式、错误处理、命名约定
9. 扩展规划

### SCHEMA.md（初始化按需：有数据库）

区块结构（全文为生成区，⚠️ 决策文件）：
1. 数据概览
2. 表关系 — 结构化文本：`tableA 1:N tableB（通过 tableB.field_name）`
3. 表结构 — 每张表一个四列表格（字段、类型、约束、说明）
4. 索引
5. 枚举/常量
6. 数据约定 — 主键策略、时间字段、软删除、字符编码
7. 迁移记录
8. 🔒 规则

### DESIGN.md（初始化按需：有界面）

区块结构：
- Part 1 视觉规范（🔓 生成区，⚠️ 决策文件）：设计风格、色彩体系（主色/中性色/语义色）、字体、间距系统、圆角、组件规范
- Part 2 交互设计（🔓 生成区，⚠️ 决策文件）：页面结构（文本描述布局）、页面流（文本描述跳转路径）、状态与反馈（加载态/空状态/错误处理）、动效规范、响应式
- 🔒 规则

### ISSUES.md（开发中按需：遇到问题时）

区块结构：
1. 🔓 **📊 快速状态** — 头块（详见上方「📊 快速状态 头块规范」），字段：活跃模块 / 起读位置 / 状态计数 / 阻塞 MVP / 更新
2. 🔓 🔴待处理 / 🟡处理中 / 🟢已修复 / ✅已验证 — 四个状态分区
3. 🔓 历史归档 — 已验证超 10 条后移入
4. 🔒 规则 — 锁定内容：编号格式（类型-模块-序号）、六种类型（BUG/DEBT/QUESTION/LIMIT/PERF/FEAT）、四态流转（🔴→🟡→🟢→✅）、不同类型必填字段（BUG 要复现步骤、PERF 要指标、QUESTION 要方案列表、FEAT 要用户故事）、所有 issue 必须有「优先级」P0/P1/P2、归档机制、**📊 快速状态 重建算法**

注：旧版的 `## 统计概览` 区块已被「📊 快速状态」头块完全取代（新头块包含了状态计数 + 更多信息）。

### DECISIONS.md（开发中按需：遇到选型时）

区块结构：
1. 🔓 **🔴 不可逆决策清单** — 顶部禁区提醒块（详见上方「📊 快速状态 头块规范」中的 DECISIONS 部分），列出所有 🔴 不可逆决策 + 数量 + 更新时间
2. 🔓 全局决策 + 各模块决策 — 每个区块下分 ✅已确定 / 💬待讨论 / 🗑️已废弃
3. 🔒 规则 — 锁定内容：ADR 编号格式（ADR-模块-序号）、三态分区、废弃留痕机制、不可逆程度必填（🟢易调整/🟡有成本/🔴不可逆）、禁止推翻🔴不可逆决策、**🔴 不可逆决策清单 重建算法**

### PROGRESS.md（开发中按需：跨 session 续接时）

区块结构：
1. 🔓 **📊 快速状态** — 头块（详见上方「📊 快速状态 头块规范」），字段：阶段 / 当前里程碑（含进度%）/ 阻塞计数 / 下次从 / 更新
2. 🔓 项目状态 — 🟡规划中 / 🔵开发中 / 🟢已上线 / 🔴已暂停
3. 🔓 最近完成 — 每完成一个 TODO 项追加
4. 🔓 当前阻塞 — 从 ISSUES 中提取阻塞项（如 ISSUES.md 存在）
5. 🔓 里程碑 — 从 SPEC 的 MVP 范围推导
6. 🔓 Session 日志 — 每次 session 结束写入
7. 🔒 规则 — 锁定内容：最近完成追加规则、超 20 条归档到 Session 日志、阻塞更新、里程碑状态、Session 日志三要素（完成了什么/遇到什么问题/下次从哪继续）、**📊 快速状态 重建算法**

## 执行流程

1. 读取 SPEC.md，提取项目信息、模块列表、技术栈、数据需求、产品形态
2. 判断第二层文件的生成条件：
   - 模块 ≥ 3？→ 生成 ARCHITECTURE.md
   - 有数据库？→ 生成 SCHEMA.md
   - 有界面？→ 生成 DESIGN.md
   - 条件不确定时询问用户
3. 按以下顺序生成文件：
   - CLAUDE.md（含第三层文件的创建规则和模板）
   - TODO.md
   - ARCHITECTURE.md（如需要）
   - SCHEMA.md（如需要）
   - DESIGN.md（如需要）
4. 做一次交叉校验：
   - CLAUDE.md 的模块缩写表和 SPEC 的模块列表一致
   - ARCHITECTURE.md 的模块依赖和 SPEC 的模块描述一致（如生成了 ARCHITECTURE.md）
   - TODO.md 的任务覆盖了 SPEC 的 MVP 范围
5. 将所有文件放在项目根目录，创建 docs/spec/ 空目录
6. 告知用户：ISSUES.md、DECISIONS.md、PROGRESS.md 将在开发过程中按需自动创建
