# spec-gen — 产品需求文档生成 Skill

基于标准化模板生成 SPEC.md，确保每个项目的需求文档结构一致、信息完整、LLM 可靠解析。

---

## 触发条件

用户在讨论完产品需求后，希望将需求整理为结构化文档。

## 输入

当前对话中讨论过的产品需求信息。如果信息不完整，主动向用户补充询问以下必填项：
1. 项目名和一句话描述
2. 目标用户是谁
3. 核心功能模块有哪些
4. 技术栈选型
5. MVP 范围（做什么、不做什么）

## 输出

一个 SPEC.md 文件，放置在项目根目录。

---

## SPEC.md 模板规范

### 文件头部

```markdown
# SPEC.md

<!-- 区块权限声明
| 区块 | 类型 | 说明 |
|------|------|------|
| 全文 | 🔓 生成区 | 根据讨论内容填写 |
| 拆分规则 | 🔒 锁定区 | 超过 300 行时触发拆分 |
-->
```

### 必须包含的区块（按顺序）

#### 1. 项目概述
必填字段：项目名、目标用户、核心价值（一句话）、产品形态

#### 2. 功能模块
每个模块必须包含以下字段：
- **功能描述**：这个模块做什么
- **用户故事**：格式为「作为 {{角色}}，我希望 {{行为}}，以便 {{目的}}」
- **核心交互**：用箭头描述主流程，如「选择文件 → 字段映射 → 预览 → 确认导入」
- **输入/输出**：明确数据进出
- **边界条件与异常处理**：列出至少 2 个

#### 3. 非功能需求
必须覆盖：性能要求、兼容性、安全性、可访问性（可标注"MVP 不处理"）

#### 4. 数据需求
用结构化文本描述核心数据实体关系，格式为：`EntityA 1:N EntityB（通过 field_name）`。如项目有 SCHEMA.md，此处写概述并引用。

#### 5. 技术约束
按前端/后端/数据库/部署分列，明确"必须使用"和"不能使用"的技术。

#### 6. MVP 范围
分为两个子区：
- **包含**：v1.0 要做的功能，每条一行
- **不包含（后续版本）**：明确列出不做的功能，防止开发时超范围

#### 7. 术语表
表格格式，统一项目内用词。至少包含项目中出现的所有领域专有名词。

### 文件尾部（拆分导航区）

```markdown
<!-- 🔒 START 锁定区：拆分导航 -->
<!-- 拆分导航区：SPEC 被拆分后激活
## 模块文档
- [模块一](docs/spec/模块一.md)
- [模块二](docs/spec/模块二.md)
-->
<!-- 🔒 END 锁定区：拆分导航 -->
```

---

## 写作规则

1. **LLM 可读性优先**：不使用 ASCII 图，所有关系用结构化文本描述
2. **不写模糊表述**：禁止"等"、"相关"、"一些"，必须具体
3. **用户故事格式强制**：每个模块必须有至少一条用户故事
4. **MVP 边界必须明确**：如果用户没说 MVP 范围，主动询问
5. **术语表必须覆盖全部领域术语**：扫描全文，所有非通用技术词都要入表
6. **超过 300 行时提醒拆分**：提醒用户按模块拆分到 docs/spec/ 下，SPEC.md 保留概述和导航链接

---

## 执行流程

1. 回顾当前对话中讨论过的所有需求信息
2. 检查必填项是否完整，缺失的向用户补充询问
3. 按模板结构组织内容
4. 生成 SPEC.md 文件
5. 检查行数，如超过 300 行则提醒用户拆分
