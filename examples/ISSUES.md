## 规则
1. 编号格式：类型-模块-序号（如 BUG-AUTH-001）
2. 类型：BUG / DEBT / QUESTION / LIMIT / PERF / FEAT
3. 模块名使用项目内约定缩写，全局性问题用 GLOBAL
4. 序号在同类型-同模块下递增，不跳号
5. 状态流转：🔴待处理 → 🟡处理中 → 🟢已修复 → ✅已验证
   - 开始修复 → 整条 issue 从「🔴待处理」区移动到「🟡处理中」区
   - 代码修复完成 → 移动到「🟢已修复」区，追加「修复日期」和「修复说明」
   - 用户确认验证 → 移动到「✅已验证」区，追加「验证日期」
6. **状态变更或任何 body 修改后，必须按下方「📊 快速状态 重建算法」整体重写顶部「📊 快速状态」区块**
7. 不同类型的必填字段：
   - BUG：复现步骤、期望行为、实际行为 必填
   - PERF：性能指标（当前值 vs 目标值）必填
   - QUESTION：可选方案列表 必填
   - FEAT：用户故事 必填
   - DEBT / LIMIT：描述即可
   - 所有 issue 都应有「优先级」字段（P0/P1/P2）
8. ✅已验证 的 issue 超过 10 条时，归档到「历史归档」区

### 📊 快速状态 重建算法

修改本文件 body（任一状态分区或归档区）后，必须按以下步骤**整体重写**顶部「📊 快速状态」区块（禁止增量修改字段）。

**输入**：当前 ISSUES.md 的 body 部分
**输出**：5 行的「📊 快速状态」块

**步骤**：
1. `N_red` = 「## 🔴 待处理」区下 issue 的数量
2. `N_yellow` = 「## 🟡 处理中」区下 issue 的数量
3. `N_green` = 「## 🟢 已修复」区下 issue 的数量
4. `N_check` = 「## ✅ 已验证」区下 issue 的数量
5. `MODULES` = 扫描「🔴 待处理」+「🟡 处理中」所有 issue 的「模块」字段 → 去重 → 字母升序 → 用反引号包裹每个值后用 `, ` 拼接
   - 若两区都为空 → `MODULES` = `（无）`
6. `START_AT` 取值规则：
   - 若「🔴 待处理」非空 → `START_AT` = 首条 issue 的编号
   - 否则若「🟡 处理中」非空 → `START_AT` = 首条 issue 的编号
   - 否则 `START_AT` = `（无活跃 issue）`
7. `BLOCK_MVP` 取值规则：
   - 扫描「🔴 待处理」+「🟡 处理中」所有 issue 的「优先级」字段
   - 若任一为 `P0` → `BLOCK_MVP` = `是`
   - 否则 `BLOCK_MVP` = `否`
8. `DATE` = 今天的日期（`YYYY-MM-DD` 绝对格式）

**填入顶部「📊 快速状态」区块**（按以下模板原样输出，将 `{var}` 替换为算出的值）：

````markdown
- **活跃模块**: {MODULES}
- **起读位置**: `{START_AT}`
- **状态计数**: 🔴 `{N_red}` / 🟡 `{N_yellow}` / 🟢 `{N_green}` / ✅ `{N_check}`
- **阻塞 MVP**: `{BLOCK_MVP}`
- **更新**: `{DATE}`
````
